# Accessibility Testing Architecture

This document describes the testing structure generated by the `ui-library` generator. It explains the design decisions, what the shared helpers cover, what they intentionally skip, and how they fit into the broader accessibility testing pyramid.

## Overview

The generator produces three test layers per component:

| File | Layer | Purpose |
|------|-------|---------|
| `<name>.component.spec.ts` | Unit (POUR) + Screen Reader | POUR-categorized: component creation, harness access, WCAG audits (Perceivable, Understandable), best-practice audits, virtual screen reader (Robust) |
| `integration/<name>.screen-reader.spec.ts` | Integration | Real screen reader testing via Guidepup + Playwright (VoiceOver / NVDA) |

### POUR structure (unit spec)

**What is POUR?** POUR is the acronym for the four foundational principles of WCAG: **P**erceivable, **O**perable, **U**nderstandable, **R**obust. They ensure content is presentable to users, operable via keyboard and other means, understandable (readable and predictable), and robust with assistive technologies. The spec includes a short description before the POUR block and before each principle block; WCAG compliance and screen reader tests live in sibling blocks below.

```
describe('ComponentName')
├── describe('POUR')                    ← short description of what POUR is
│   ├── describe('Perceivable')        ← description only; tests in Regression - WCAG Compliance
│   ├── describe('Operable')           ← description + should create, accessible via harness
│   ├── describe('Understandable')     ← description + should follow best practices
│   └── describe('Robust')             ← description only; tests in Screen Reader block
├── describe('Regression - WCAG Compliance')
│   ├── it('should be WCAG A compliant')     ← expectWcagACompliant (Perceivable)
│   └── it('should be WCAG AA compliant')    ← expectWcagAACompliant (Perceivable)
└── describe('Screen Reader')
    └── it('should have meaningful spoken output')  ← @guidepup/virtual-screen-reader (Robust)
```

| POUR principle | Description | Where tested |
|----------------|-------------|--------------|
| **Perceivable** | Information and UI presentable (e.g. text alternatives, contrast) | Regression - WCAG Compliance |
| **Operable** | UI operable (keyboard, focus, navigation) | POUR > Operable |
| **Understandable** | Content readable and predictable, input assistance | POUR > Understandable |
| **Robust** | Works with assistive tech (valid semantics) | Screen Reader block |

### Integration spec

```
describe('ComponentName - VoiceOver')  ← macOS only
├── it('should have a meaningful reading order')
└── it('should announce component content')

describe('ComponentName - NVDA')       ← Windows only
├── it('should have a meaningful reading order')
└── it('should announce component content')
```

The regression tests delegate to dedicated helpers from `@angular-comic-store/test-helpers`:

```typescript
// libs/util/test-helpers/src/lib/axe-core.functions.ts

async function runAxe(fixture: ComponentFixture<unknown>, tags: string[]) {
  fixture.detectChanges();

  const results = await axe.run(fixture.nativeElement, {
    runOnly: { type: 'tag', values: tags },
  });

  expect(results.violations).toEqual([]);
}

export async function expectWcagACompliant(fixture: ComponentFixture<unknown>) {
  await runAxe(fixture, ['wcag2a', 'wcag21a', 'wcag22a']);
}

export async function expectWcagAACompliant(fixture: ComponentFixture<unknown>) {
  await runAxe(fixture, ['wcag2aa', 'wcag21aa', 'wcag22aa']);
}

export async function expectWcagAAACompliant(fixture: ComponentFixture<unknown>) {
  await runAxe(fixture, ['wcag2aaa']);
}

export async function expectBestPracticesCompliant(fixture: ComponentFixture<unknown>) {
  await runAxe(fixture, ['best-practice']);
}
```

## Available Helpers

The helpers are split by conformance level so tests can target exactly the level they need:

| Function | axe-core Tags | Use Case |
|----------|---------------|----------|
| `expectWcagACompliant` | `wcag2a`, `wcag21a`, `wcag22a` | Minimum conformance -- Level A only |
| `expectWcagAACompliant` | `wcag2aa`, `wcag21aa`, `wcag22aa` | Legal conformance target -- Level AA |
| `expectWcagAAACompliant` | `wcag2aaa` | Aspirational -- Level AAA |
| `expectBestPracticesCompliant` | `best-practice` | Non-WCAG patterns widely accepted as important |

The generator uses `expectWcagAACompliant` and `expectBestPracticesCompliant` by default. Components with stricter requirements can additionally call `expectWcagAAACompliant` or narrow the scope to `expectWcagACompliant`.

## Why Separate Functions?

**Granular conformance targeting.** Different components may have different conformance goals. A marketing page might target AAA while an internal tool targets AA. Splitting by level lets each test express its intent clearly.

**Independent failure signals.** When AA and best-practice checks run in separate assertions, a failing best-practice rule doesn't obscure a WCAG violation (and vice versa). CI output immediately tells you which conformance level broke.

**Composable.** Tests can call any combination of helpers. The generator defaults to AA + best-practice, but nothing prevents adding A or AAA checks alongside.

## Default Generator Coverage: AA + Best Practices

**AA is the legal standard.** WCAG Level AA is the conformance target for most accessibility regulations (ADA, EN 301 549, Section 508). Level A is a subset of AA, so passing AA implicitly covers A.

**Best Practices fill the gaps.** The `best-practice` tag catches patterns not tied to a specific WCAG criterion but widely accepted as important: landmark regions, heading order, skip links, tabindex usage, and meta viewport restrictions.

## What the Default Tests Cover

| Function | What It Catches |
|----------|----------------|
| `expectWcagAACompliant` | All WCAG 2.0/2.1/2.2 Level AA rules: missing alt text, empty buttons, form labels, ARIA roles, color contrast, focus visible, reflow, text spacing, target size, focus not obscured, accessible authentication |
| `expectBestPracticesCompliant` | Landmark structure, heading hierarchy, skip navigation, tabindex misuse, meta viewport restrictions, autocomplete attributes |

## What the Default Tests Intentionally Skip

axe-core has additional rule tags that are **not** included by default:

| Tag | Why Skipped | Available Via |
|-----|------------|---------------|
| `wcag2aaa` | Level AAA is aspirational, not a conformance target. Most criteria (enhanced contrast 7:1, sign language, no timing) cannot be meaningfully validated at the component level. | `expectWcagAAACompliant` |
| `cat.aria` | ARIA-specific rules overlap heavily with WCAG AA. Rules not already covered are edge cases better caught in integration tests. | -- |
| `cat.forms` | Form accessibility rules are covered by WCAG AA tags (labels, error identification). Category tags duplicate what the WCAG tags already run. | -- |
| `cat.keyboard` | Keyboard navigation rules in axe-core are limited to static analysis. Real keyboard testing is done via Component Harness and Guidepup. | -- |
| `cat.name-role-value` | Semantic role rules are a subset of WCAG 4.1.2, already included in WCAG AA tags. | -- |
| `experimental` | Unstable rules that may produce false positives. Not suitable for CI pipelines. | -- |
| `wcag2a-obsolete` | Deprecated WCAG mappings retained for backward compatibility. Running them adds noise without value. | -- |

## What axe-core Can and Cannot Catch

axe-core provides **automated** accessibility testing and can detect approximately **57% of WCAG issues**. The remaining issues require manual testing or assistive technology validation.

| Automated (axe-core) | Requires Manual Testing |
|----------------------|------------------------|
| Missing alt text | Quality/accuracy of alt text |
| Color contrast ratios | Meaning conveyed by color alone |
| Missing form labels | Label clarity and helpfulness |
| Invalid ARIA attributes | Correct ARIA usage in context |
| Duplicate IDs | Logical reading order |
| Missing document language | Cognitive load and plain language |
| Focus order (partial) | Complete keyboard workflow |
| Landmark structure | Screen reader announcement quality |

## Where This Fits in the Testing Pyramid

```
┌─────────────────────────────────────────────────────┐
│  Layer 4: Scaffolding (Nx Generators)               │  ← You are here (generator)
│  Auto-generate accessible boilerplate               │
├─────────────────────────────────────────────────────┤
│  Layer 3: Story/E2E (Storybook Play Fns)            │
│  Full interaction paths, themes, tokens             │
├─────────────────────────────────────────────────────┤
│  Layer 2: Integration (Guidepup + Playwright)       │  ← Integration specs land here
│  Real screen reader order, announcements            │
├─────────────────────────────────────────────────────┤
│  Layer 1: Unit (Vitest + CDK Harness + axe-core     │  ← POUR-categorized unit + virtual SR
│           + Virtual Screen Reader)                  │
│  Component creation, harness, WCAG, spoken output   │
└─────────────────────────────────────────────────────┘
```

- **Unit tests** (`*.spec.ts` POUR block) verify component creation, harness access, WCAG compliance (Perceivable, Understandable), and virtual screen reader output (Robust). Fast, run on every save; axe audits run as a CI gate.
- **Virtual screen reader tests** (`*.spec.ts` Screen Reader block, sibling to POUR) use `@guidepup/virtual-screen-reader` to verify spoken output in-process without a real screen reader. Fast enough for CI.
- **Integration tests** (`integration/*.screen-reader.spec.ts`) use Guidepup + Playwright to verify reading order and announcements with real screen readers (VoiceOver on macOS, NVDA on Windows).
- **Story/E2E tests** (Storybook + Playwright) validate full user interaction paths across themes and viewports.
- **Scaffolding** (this generator) ensures every new component starts with all test layers baked in.

## Virtual Screen Reader (Unit Layer)

The generator adds a `describe('Screen Reader')` block (sibling to POUR) to the unit spec using `@guidepup/virtual-screen-reader`. This library simulates screen reader behavior in-process (no real assistive technology required), making it fast enough to run alongside unit tests.

### Virtual Screen Reader Helpers

| Function | Purpose |
|----------|---------|
| `startVirtualScreenReader(container)` | Start the virtual screen reader on an HTML element |
| `stopVirtualScreenReader()` | Stop the virtual screen reader and release resources |
| `getSpokenPhrases()` | Navigate to end of document, return all spoken phrases |
| `expectSpokenPhrasesToContain(phrases, expected)` | Assert expected fragments appear in order |

### What Virtual Screen Reader Tests Catch

- Empty or missing text content (spoken output is empty)
- Incorrect ARIA labels that produce unexpected announcements
- Logical reading order problems visible in the phrase sequence
- Missing semantic structure (headings, landmarks) that screen readers rely on

### What Virtual Screen Reader Tests Cannot Catch

The virtual screen reader operates on the accessibility tree but does not replicate vendor-specific behavior. Real screen readers (VoiceOver, NVDA, JAWS) each have quirks in how they interpret ARIA, handle live regions, and announce dynamic content. The integration layer (Guidepup + Playwright) covers those gaps.

## Guidepup Integration Tests

The generator also produces `integration/<name>.screen-reader.spec.ts` for real screen reader testing via Playwright. These tests run against a Storybook instance and use:

- **VoiceOver** (macOS only) via `voiceOverTest` from `@guidepup/playwright`
- **NVDA** (Windows only) via `nvdaTest` from `@guidepup/playwright`

Integration tests are configured via `playwright.screen-reader.config.ts` at the workspace root, which scans `./libs` for `*.screen-reader.spec.ts` files.

### Integration Helpers

| Function | Source | Purpose |
|----------|--------|---------|
| `navigateAndCollectPhrases(screenReader)` | `guidepup.helpers.ts` | Navigate and collect all spoken phrases |
| `assertReadingOrder(phrases, expected)` | `guidepup.helpers.ts` | Assert reading order of spoken phrases |
| `collectSpokenPhraseLog(screenReader)` | `guidepup.helpers.ts` | Get the full spoken phrase log |

All helpers are exported from `@angular-comic-store/test-helpers`.

## Component Harness

Each generated component also includes a CDK `ComponentHarness` and a smoke test (`should be accessible via component harness`). The harness provides:

- Stable selectors decoupled from DOM structure
- Async-aware interaction primitives (click, sendKeys, focus)
- A foundation for keyboard navigation and focus management tests that go beyond what axe-core audits

Harness tests complement axe-core by verifying **behavioral** accessibility (keyboard operability, focus trapping, arrow key navigation) rather than just **structural** accessibility (ARIA, contrast, labels).
